## 💡 なぜ作るか（背景・課題）

小林モータースには現在、**バラバラの予定管理ツール**が混在している：

```
現状の二重・三重管理：
├ チームウェア（ファイルサーバー付属）= 全社・受付・車販・整備の大枠予定
│   └ 過去7年・12,387件のデータ蓄積あり
├ 整備のホワイトボード（紙）= 入庫予定カレンダー（週次）
├ 整備の当日スケジュール（紙）= その日の作業段取り
└ 車販のホワイトボード（紙）= 商談・展示・納車予定
```

これを段階的に各 Flow アプリに統合：

```
将来：
├ CarFlow      … 車販のホワイトボード・納車予定（既に統合済み）
├ PitFlow      … 整備のホワイトボード2枚を統合（次の本開発）
├ MHS          … 社内カレンダー（チームウェア代替）＋全アプリ予定の俯瞰
└ 紙           … ゼロ化（全部電子化）
```

**MHS が解決する核心問題**：
1. **チームウェアからの完全移行先がない**（社内予定の置き場所）
2. **「会社全体の今週・今月の動き」を1画面で見たい**（経営判断・配車計画用）
3. **週次の車検段取り**を勘でやってる → 漏れ・無駄移動・スタッフ過負荷が発生
4. **「誰がいつ何できる」のデータが頭の中**にしかない → 人に依存・属人化

## 🧩 主要機能

### A. 社内カレンダー（編集機能・本丸の入れ物）

チームウェアの**完全代替**として、社内予定のCRUDを担う。

- **3階層**：全社 / 部署（車販・受付・整備） / 個人
- **担当プレフィックス文化を尊重**：チームウェア由来の「専務：千葉銀行様来社」「整備：MTG」形式を継承
  - 担当者選択UIを設けて自動で「担当：内容」表示
- **定期予定テンプレート**：「ノー残DAY」「月例報告会」「自賠精算」等の繰り返しを1回設定で済むように
- **タグ**：顧客来店・支払日・休日・MTG・出張・その他
- **添付**：URL・ファイル・関連アプリの予定ID（CarFlow案件等へジャンプ）

### B. 統合ビュー（俯瞰閲覧専用・本丸の見方）

**各 Flow アプリの予定を Firestore から読み取って重ねる**。編集は各アプリで完結＝ここでは見るだけ。

- **レイヤー切替**：「CarFlow納車」「PitFlow整備」「StockFlow納品」「社内全社」「社内部署」をチェックボックスでON/OFF
- **色分け**：CarFlow=青・PitFlow=緑・StockFlow=紫・社内=黒・締切=赤
- **他アプリの予定をクリック**→そのアプリの該当画面へジャンプ（編集はそっちで）

### C. 多軸カレンダー（核心の見せ方）

普通のカレンダーは「時間×予定」の1軸だけ。御社の意思決定は「時間×**誰**×**どの車**×**どこ**」の交差で決まるので、**軸を切り替えられる**：

| 軸 | 見え方 | 用途 |
|---|---|---|
| 日付軸（月/週/日） | 普通のカレンダー | 慣れた見方・全員向け |
| **メンバー軸** | 縦=社員、横=時間 | 「誰が今何してて・いつ空く」 |
| **車両軸** | 縦=預かり車両、横=時間 | 「各車の入庫→整備→車検→返車レーン」 |
| **場所軸** | 縦=行き先（陸運局A/B/部品屋）、横=時間 | 「同方面のまとめ取り提案材料」 |

**1つの予定はどの軸から見ても同じデータ**（Excelのピボット感覚）。

### D. ダッシュボード／朝礼カード

朝礼で全員で見る用の**「今日」「明日」専用画面**。

- 今日のイベント一覧（時系列）
- 担当者別の今日の動き
- 警告：「今日2件納車予定が逆方向」「明日の車検配車未定」
- 今日休む人・外出する人

### E. 印刷・モバイル

- **A3週カレンダー1枚で全部入る印刷**（壁貼り用・モノクロでも読める設計）
- **スマホは「今日と明日だけ」の縦スクロール版**（現場用・潰れない）
- 担当者ごとの自分の予定だけ出すフィルタ（現場スタッフの常用画面）

### F. AI配車・段取り提案（Phase 4・Claude API）

データが揃った段階で投入。**MHS の本丸機能**。

- **週次車検段取り提案**：
  - 「水曜のABC陸運局へ田中（MT可）×車両X＋ついでにDEF部品屋に寄れる」
  - 抱き合わせ最適化・スタッフ負荷分散・期日厳守
- **シフト穴埋め提案**：誰が代われる、スキル足りる
- **来店ラッシュ対応**：受付混雑予測＋応援配置
- **理由つきの強い提案＋ゆうたが微調整**（数学的最適解ではない）

## 🔐 役割分担の鉄則（2026-05-29 ガチMTGで確定）

**「編集は各アプリで完結／MHSは俯瞰閲覧専用＋社内予定の編集だけ」**

| 予定の種類 | 編集場所 | MHSでの扱い |
|---|---|---|
| 整備の入庫予定・当日工程 | **PitFlow** | 読み取って統合ビューに重ねる |
| 車販の納車予定・付箋 | **CarFlow** | 読み取って統合ビューに重ねる |
| 物販の納品・請求予定 | **StockFlow** | 読み取って統合ビューに重ねる |
| **社内予定（全社・部署・個人）** | **MHS** | 編集可能（チームウェア代替） |
| MoneyFlow の支払日・締日 | MoneyFlow | 読み取って統合ビューに重ねる |
| CoreMembers の研修・1on1予定 | CoreMembers | 読み取って統合ビューに重ねる |

**なぜこうするか**：
- 業務文脈を壊さない（納車予定は CarFlow の車両に紐づいてるべき）
- 編集の入口を分散すると、二重管理になる（チームウェアの轍を踏まない）
- 「経営判断する人＝ゆうた・専務・社長」だけが俯瞰ビューを使う
- 現場スタッフは自分のアプリで業務する＝学習コスト分散

```
│   ├ relatedAppRef: {app, id}          関連アプリの予定ID（あれば）

│
├ scheduleViewConfig/{uid}             ← 個人ビュー設定
│   ├ activeLayers: ['carflow','pitflow',...]
│   ├ defaultAxis: 'date'|'member'|'vehicle'|'place'
│   └ filters
│

│   ├ status                            'work'|'off'|'half' 等
│   └ source: 'csv'|'manual'|'linked'   取込手段（CSV/Excel・手入力・既存シフトアプリ連携）

    ├ targetWeek

    ├ reasoning                         理由文
    ├ acceptedByUid                     誰が採用したか
    └ feedback                          採用/却下・修正内容
```
